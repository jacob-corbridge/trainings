Foundation

The vocabulary you'll hear

Terms that appear constantly in Fabric documentation. Park here as long as needed.
Tenant

Your organization's top-level Microsoft cloud space. Stotz Equipment is one tenant. Everyone in the company shares it. Permissions, billing, and policies are all managed at this level.

ETL (Extract, Transform, Load)

The process of pulling data from a source, reshaping it, and putting it somewhere useful. "Extract from Salesforce, transform into clean rows, load into the warehouse." Most data work is ETL with extra steps.

Schema

The structure of your data. What columns exist, what types they are (number, text, date), what's required, what's optional. A schema is the contract that says "every row in this table will look like this."

Stored Procedure

A saved chunk of SQL code that runs against a database. Instead of writing the query every time, you save it once and call it by name. Common for complex business logic that runs on a schedule.

SQL / T-SQL

SQL is the standard language for querying databases. T-SQL is Microsoft's flavor with extra features (variables, control flow, error handling). If you've written queries in SQL Server, you've written T-SQL.

Spark / PySpark

Spark is an engine for processing huge datasets across many computers at once. PySpark is the Python interface to Spark. You write Python; Spark distributes the work. Used in data engineering for transformations too big for a single machine.

Parquet

A file format for tabular data. Unlike CSV, it stores columns separately and compressed. A query that needs three columns from a thousand-column table only reads three column files, not the whole row.

ACID transactions

The guarantee that database updates either fully succeed or fully fail, with no half-states. If a job updates 1,000 rows and fails on row 500, ACID ensures none of the 500 already-changed rows are kept.

The vocabulary you'll hear

Continued. The harder ones get a paragraph.
Git

A version control system. Tracks every change to a set of files over time. Two people can edit the same file separately and Git helps merge their work. Originally built for source code; now used for SQL, dataflows, configurations.

GitHub

A website that hosts Git repositories. The repo lives there, your team's changes sync to it, and it gives you the web UI for reviewing changes, leaving comments, and approving updates before they go live.

CI/CD

Continuous Integration / Continuous Deployment. Shorthand for: when someone changes code in Git, automated tests run, and if they pass, the change is deployed to production. The point is to ship small changes safely instead of giant scary releases.

S3 / GCS

Amazon S3 is AWS's cloud file storage. Google Cloud Storage (GCS) is Google's. Both are common places to find data Fabric might need to read. ADLS is Microsoft's equivalent.

SaaS API

A way to read or write data in a SaaS product programmatically. Salesforce, HubSpot, ServiceNow, Workday, Stripe — all of them expose APIs that pipelines can call to pull data into Fabric.

Event Stream

A continuous flow of small messages: an IoT sensor emitting a reading every second, a website logging clicks, a payment system reporting transactions. The data never stops; tools have to handle it as it arrives, not in batches.

Notebook

A document that mixes code, written explanations, and the output of running that code. Used heavily in data science and engineering. Think of it as a hybrid of a Word document and a script.

KQL (Kusto Query Language)

A query language Microsoft built for time-series and log data. Used in Real-Time Intelligence. SQL-like but optimized for asking "what happened in the last 5 minutes" against huge volumes of events.

Context

What Microsoft Fabric replaces

Fabric absorbed several Azure tools your team may have heard of. Here's where they fit.
The old way

Stitched-together Azure services

Building an analytics stack in Azure used to mean buying and connecting separate products:

  • Azure Data Lake Storage (ADLS) for raw file storage
  • Azure Synapse Analytics for the SQL warehouse
  • Azure Databricks for Spark-based engineering and data science
  • Azure Data Factory for moving and orchestrating data
  • Power BI for visualization

Each had its own permissions, its own bill, its own storage. Data had to be copied between them.

The Fabric way

One SaaS platform

Fabric collapsed those products into a single service:

  • OneLake replaces standalone ADLS for analytics workloads
  • Fabric Warehouse covers what Synapse SQL did
  • Fabric Data Engineering covers what Databricks did (Spark notebooks)
  • Fabric Data Factory covers what Azure Data Factory did
  • Power BI is now a workload inside Fabric

One bill. One permission model. One storage layer. No copies between products.

About Azure Data Factory specifically: it still exists and many companies still use it for non-Fabric Azure workloads. It is not deprecated. But Microsoft is steering all new development toward Fabric Data Factory, and the two share most of the same concepts (pipelines, copy activities, triggers).
Concept

What Microsoft Fabric is

One SaaS platform for analytics. One data lake underneath. Many workloads on top.

Fabric is a single product that handles every step of working with data: getting it from sources, transforming it, storing it, analyzing it, and visualizing it.

The architectural insight is that all of those steps share one storage layer called OneLake. A pipeline writes a table. A notebook reads that same table without exporting. A Power BI report visualizes it without importing. No data movement between Fabric services.

This is the difference from the old Azure stack, where every product had its own storage and data had to be copied between them.

OneLake

Foundation
"The OneDrive for data." One logical data lake, automatic with every Fabric tenant.

One per tenant

Stotz Equipment has one OneLake. You can't create more, can't delete it. Every workspace stores its data there automatically. No infrastructure to provision.

Open file format

Tables stored as Delta Parquet, an open standard. Fabric, Spark, Databricks, even Snowflake can read the same files. No proprietary lock-in.

Shared by every workload

Lakehouses, Warehouses, Power BI, notebooks — all read and write OneLake. The same table is accessible to every tool without copies or exports.

How OneLake is organized:   Tenant → Workspaces → Items (Lakehouses, Warehouses, Reports) → Files & Tables

Workspaces are the security and collaboration boundary. Permissions are granted per workspace. Within a workspace, items are the things you create.

Source: Microsoft Learn · learn.microsoft.com/fabric/onelake
OneLake feature

Shortcuts

Pointers, not copies. The most useful OneLake feature for multi-source work.

A shortcut is a reference inside OneLake that points to data living somewhere else. The data isn't moved. It isn't copied. OneLake just records "data X lives over there, treat it like it's here."

When a query runs against a shortcut, Fabric reads through to the actual source at query time.

A concrete example

Your sales team's CSV exports land in an Amazon S3 bucket. Instead of copying them into OneLake nightly:

  • Create a shortcut from your Lakehouse pointing at the S3 bucket
  • The CSV files appear in your Lakehouse Files section as if they lived there
  • A Power BI report can query them. A notebook can transform them.
  • S3 stays the source of truth. No sync job to maintain.

What shortcuts can point at

Inside OneLake

Other workspaces, other Lakehouses. Lets one team reference another team's curated tables.

External storage

ADLS Gen2 (Microsoft), S3 (AWS), GCS (Google). The cloud where the data lives doesn't matter.

Dataverse

This is what Link to Fabric uses. Dataverse data appears in OneLake without leaving Dataverse.

Watch the egress: reads against an S3 shortcut hit S3 every time. AWS charges egress fees. Frequent queries add up.
Foundation

What Delta Parquet actually is

Two ideas stacked. Parquet is the file format. Delta is the transaction layer.

Parquet

A file format optimized for analytics. The key difference from CSV: Parquet stores data column by column, not row by row.

If a table has 100 columns and your query asks about 3 of them, Parquet only reads those 3. CSV would have to read every byte of every row.

It's also compressed. A Parquet file is typically 70-90% smaller than the equivalent CSV.

Delta

A layer on top of Parquet. A Delta table is a folder of Parquet files plus a transaction log that tracks every change.

The log is what makes a folder of files behave like a database table. Updates, deletes, schema changes, time travel — all of it works because Delta records what happened and when.

What Delta gives you
01

ACID transactions

An update either fully succeeds or fully rolls back. No half-updated tables.

02

Time travel

Query the table as it looked yesterday, last week, or at a specific transaction. The log keeps history.

03

Schema enforcement

Bad data gets rejected at write time, not silently appended. The table's column types are a contract.

04

Concurrent writers

Multiple jobs can write to the same table without corrupting state.

Why it matters: Delta is the universal table format inside Fabric. When someone says "the table" in Fabric, they usually mean a Delta table. The Lakehouse exposes them, the Warehouse creates them under the hood, and Power BI can read them through DirectLake mode without importing.

Semantic Models

Core tool
The business logic layer. Where raw tables become "Revenue," "Customer," and "Active Account."

A semantic model (formerly called a Power BI dataset) is a layer on top of your tables that defines relationships, measures, and hierarchies.

It connects your fact tables to your dimension tables. It's where DAX measures like "Total Sales" or "YoY Growth" are authored. And it's where the column names users actually see in reports get standardized.

Multiple Power BI reports can share one semantic model. Define the metric once, every report calling it shows the same number.

Power Query vs DAX clarification:

Power Query (M language) reshapes data before it lands in the model — renaming columns, filtering rows, merging tables. DAX is authored in the model itself, in Power BI Desktop's data view, after the data is loaded. They're sequential, not the same thing.

The flow

Tables

Raw data in OneLake. Product, Order, Customer.

↓

Power Query

Clean, filter, reshape. Output goes into the model.

↓

Semantic Model

Relationships defined. DAX measures written here.

↓

Reports

Visuals built on the model's measures.

Power Query

Core tool
Not exclusively a Fabric tool. A transformation engine that shows up everywhere Microsoft does data.

Power Query is Microsoft's no-code/low-code data transformation engine. You point it at a source, click steps to clean and reshape data (rename columns, filter rows, pivot, merge, etc.), and it records those steps as a script in a language called M.

Same Power Query, lands data in Dataverse tables.

Tools

Analysis Services, Dynamics 365

Data Factory

Core tool
How data gets into Fabric and how transformations get scheduled.
Tool 1

Pipelines

Orchestration. The "do this, then this, then this" tool.

  • Copy Data activities to move data between sources and OneLake
  • Notebooks, stored procedures, and Dataflows can all be steps
  • Triggers: scheduled, on-demand, or event-based
  • Conditional branching, loops, and parallel execution
  • Replaces Azure Data Factory for Fabric workloads
Tool 2

Dataflows Gen2

Transformation. Power Query at Fabric scale.

  • Power Query Online interface (the same one Excel users know)
  • Connects to 100+ sources, lands output in Lakehouse, Warehouse, or KQL DB
  • Backed by Fabric SQL compute (faster than Gen1)
  • Auto-saves, supports CI/CD and Git integration as of April 2026
  • Can be invoked as a step inside a Pipeline
Rule of thumb: Dataflows Gen2 to transform. Pipelines to orchestrate. Often used together — a Pipeline runs on a schedule, calls a Dataflow Gen2 as one step, then runs a notebook as the next step.

Power BI

Core tool
The visualization layer. Now a workload inside Fabric, not a separate product.

Power BI is the same tool your team has likely used for years. The change with Fabric is administrative: Power BI workspaces are now Fabric workspaces. Premium capacity is now Fabric capacity. Reports, semantic models, and dashboards are now "items" inside Fabric alongside Lakehouses and Notebooks.

The three connection modes

Mode 1

Import

Data is copied into the model file (the .pbix). Stored compressed in memory. Queries are blazing fast because they hit RAM.

Tradeoff: data is stale until the next scheduled refresh. Models can get huge — gigabytes for big datasets.

Mode 2

DirectQuery

Queries hit the source database live, every time a visual loads. Always current.

Tradeoff: slower interactions. Heavy load on the source. Some DAX features unavailable.

Fabric-native

DirectLake

Reads Delta tables in OneLake directly at query time. Import-mode speed without a separate copy of the data.

This is the new default for Fabric work. No refresh schedule, no synchronization lag.

Lakehouse

Workload
A workspace item that organizes both files and Delta tables. The data engineer's home base.

A Lakehouse has two sides:

  • Files — unstructured or semi-structured data. CSVs, JSON, Parquet, images, anything.
  • Tables — Delta tables that can be queried with SQL.

You ingest raw data into Files, then transform it into Tables. The Tables side is automatically queryable through a SQL endpoint, even if the data was loaded by a Spark notebook.

It's the most common starting point in Fabric. If your team is exploring data without strong opinions on structure yet, you put it in a Lakehouse.

When to use a Lakehouse

Loading raw data from many sources before deciding on schema

Mixing structured tables with files (PDFs, images, JSON logs)

Spark-based transformations using Python or Scala notebooks

Building Bronze and Silver layers in a Medallion architecture

Warehouse

Workload
Same OneLake storage, but a fully transactional T-SQL surface. The SQL developer's home base.

The Warehouse is what you use when you want to work with data using T-SQL stored procedures, the way SQL developers have for decades.

Under the hood, the Warehouse stores its tables as Delta Parquet in OneLake — same format as a Lakehouse. The difference is the developer experience: full T-SQL support, ACID transactions across multiple tables in a single statement, native stored procedures, views, security with row-level filtering.

A Warehouse table is readable from a Lakehouse and vice versa. The format is shared. Choose based on who's writing the code.

When to use a Warehouse

Team is SQL-fluent and prefers T-SQL over Python or Spark

Need traditional warehouse features: stored procs, views, schemas, fine-grained permissions

Multi-table transactional updates that must commit atomically

Building the Gold layer in a Medallion architecture (curated, business-ready data)

Data Engineering

Workload
Spark notebooks for code-first transformations. Where the heavy lifting happens.

Data Engineering is the workload for writing code-based data transformations against Lakehouse tables. The interface is Jupyter-style notebooks — the same notebook concept used in data science, just pointed at production data.

You write PySpark (Python on Spark), Scala, Spark SQL, or R. Fabric runs the code on a managed Spark cluster. No infrastructure to provision.

This is what you reach for when Power Query and Dataflows Gen2 hit their limits — complex multi-step transformations, custom logic, performance tuning, processing data that's too large for the no-code tools to handle gracefully.

Notebooks can be saved, scheduled, and called from Pipelines as part of larger workflows.

When to use Data Engineering

Transformations are too complex or too large for Dataflows Gen2

You need version-controlled, code-reviewed transformation logic

Existing PySpark or Databricks code needs to run inside Fabric

Production-grade Bronze→Silver→Gold pipelines that demand reliability

Real-Time Intelligence

Workload
For data that arrives continuously instead of in batches. IoT, telemetry, financial feeds.

Real-Time Intelligence handles streaming data. Where a Lakehouse holds tables that are updated periodically by pipelines, Real-Time Intelligence handles data arriving as a continuous flow.

Three concepts you'll hear in this space:

  • Eventstream — the ingestion layer. Connects to sources like Azure Event Hubs, Kafka, IoT Hub. Routes events to destinations.
  • Eventhouse — the storage. A KQL database optimized for time-series and log data. Queryable in seconds even at billions of rows.
  • Real-Time Dashboards — visualizations that update as new events arrive.

When to use Real-Time Intelligence

Equipment telemetry from machines on a job site or facility

Application logs that need real-time alerting

Customer activity streams (clicks, transactions, support events)

Anything where "what happened in the last 5 minutes" matters

Eventhouse data also lives in OneLake, so it can be joined with batch data from Lakehouses or Warehouses.

Data Science

Workload
Machine learning lifecycle, end to end, against Lakehouse data.

Data Science is Fabric's workload for building and deploying machine learning models. The workspace is Jupyter-style notebooks backed by managed Spark, with access to the full Python ML ecosystem — scikit-learn, XGBoost, PyTorch, TensorFlow.

Fabric integrates MLflow natively. MLflow is the open-source standard for tracking experiments (which model, which parameters, which metrics) and registering trained models for deployment.

Trained models can run batch predictions directly against Lakehouse tables using a function called PREDICT. No separate serving infrastructure for batch inference.

The advantage of doing this inside Fabric: feature tables built by data engineers in the Lakehouse are immediately accessible in ML notebooks. No copying or re-ingesting data.

When to use Data Science

Predictive maintenance models against equipment telemetry

Demand forecasting using historical sales data

Customer segmentation or churn prediction

Any ML work that benefits from co-locating models and training data

Activator

Workload
The "if this, then that" layer. Watches data and triggers actions when conditions are met.

Activator (also called "Data Activator") is the workload that turns data into action. You point it at a stream or a table, define a condition, and tell it what to do when the condition is true.

The actions it can trigger include: send an email, post to Teams, call a Power Automate flow, kick off a Fabric notebook. The "Power Automate flow" hook is the bridge to the Power Platform — a data signal in Fabric can launch a workflow that updates a Dataverse record.

A concrete example:

Inventory data lands in OneLake every hour. Activator watches a stock-on-hand table. When any product drops below its reorder threshold, Activator triggers a Power Automate flow that creates a purchase request record in Dataverse and emails the relevant manager.

When to use Activator

Threshold-based alerts that trigger business actions, not just notifications

Bridging analytics insights to Power Platform automations

Replacing custom monitoring scripts with declarative rules

Anywhere Power BI dashboards are being checked manually for "is X red?"

Pattern

The Medallion Architecture

How to organize data work so it stays maintainable as it grows. Bronze, Silver, Gold.

Medallion is a structural pattern, not a Fabric feature. It's the convention that has emerged for organizing analytics data in three layers, each progressively cleaner and closer to business use. Most mature Fabric deployments follow it.

Bronze

Raw

Goal: capture source data exactly as it arrives. Don't transform. Don't filter.

  • Salesforce export, raw CSV
  • API responses, untouched JSON
  • Database mirror, byte-for-byte
  • Equipment telemetry, raw events

If something breaks downstream, you can always rebuild from Bronze.

Silver

Cleaned

Goal: data that's technically correct. Joined, deduplicated, validated.

  • Standardized column names
  • Type-corrected (dates as dates)
  • Bad rows filtered or flagged
  • Foreign keys resolved

Silver is trustworthy enough for engineers, but not for executives.

Gold

Curated

Goal: business-ready datasets. Aggregated, modeled, named in business terms.

  • Sales by region by month
  • Customer 360 (joined view)
  • Active equipment fleet metrics
  • KPI tables for dashboards

Gold tables are what semantic models and reports point at.

How this maps to Fabric tools: Bronze typically lives in a Lakehouse (raw files or tables). Silver lives in a Lakehouse or Warehouse. Gold often lives in a Warehouse for SQL teams or as curated Lakehouse tables for engineering teams. Each layer can be a separate workspace for governance and access control.
Operations

Capacity & CUs

How Fabric is billed and what to plan for.

Fabric is sold as capacity. You buy a capacity SKU (F2, F4, F8, F16, F64, etc. — the number indicates relative size) and every workload in your tenant runs against that capacity.

Capacity is measured in Capacity Units (CUs). A CU is an abstract unit of compute. Different operations consume different amounts.

  • Running a Power BI report → small CU usage
  • Refreshing a semantic model → moderate CU usage
  • Running a Spark notebook → high CU usage
  • Mirroring a database continuously → ongoing CU usage

Storage is billed separately and cheaply. The expensive part is compute.

Things to know

Smoothing

Spikes in CU usage are smoothed over a 24-hour window. A short burst over your limit doesn't immediately throttle you.

Throttling

Sustained over-use eventually throttles workloads. The Fabric Capacity Metrics app shows utilization and helps right-size.

Pause/resume

Capacities can be paused to stop billing for compute, useful for dev environments off-hours.

Power BI Premium inheritance

If you have Power BI Premium capacity, you can use it as Fabric capacity for Fabric workloads at no additional charge.

The big question

Fabric is not the Power Platform

They're sibling products. Both Microsoft. Different jobs.
Microsoft Fabric

Analytics platform

Built for moving, transforming, and analyzing data at scale. The audience is data engineers, analysts, and BI developers.

What lives here
  • OneLake (the data lake)
  • Lakehouses, Warehouses, Notebooks
  • Pipelines, Dataflows Gen2
  • Power BI reports & semantic models
Power Platform

Business apps platform

Built for building business applications and automating processes. The audience is citizen developers and business users.

What lives here
  • Dataverse (the operational database)
  • Power Apps (canvas & model-driven)
  • Power Automate (workflow automation)
  • Power Pages, Copilot Studio
The mental model: Dataverse is to the Power Platform what OneLake is to Fabric. Each platform has its own primary data store, designed for what its tools need.
Power Platform context

Canvas vs Model-driven apps

Two different philosophies for building Power Apps. Worth understanding the distinction.
Approach 1

Canvas Apps

UI first. Drag-and-drop the screen, then connect data.

  • Start with a blank canvas, design the screens you want
  • Pixel-level control over layout and styling
  • Connect to any data source (not just Dataverse)
  • Custom controls extensible via TypeScript and the Power Apps Component Framework (PCF)
  • Best for purpose-built apps with custom workflows

Use case: a field inspection app where the screen layout has to match a specific paper form.

Approach 2

Model-driven Apps

Data first. Define the tables and relationships, app generates the UI.

  • Built on Dataverse (required, not optional)
  • Auto-generated forms, views, and navigation based on table structure
  • Less design control, more data structure focus
  • Strong out-of-the-box record management, history, and security
  • Best for record-management apps that look like CRMs

Use case: a customer service tracking app where users navigate accounts, contacts, and cases.

Integration

How Dataverse and OneLake connect

Same data, different access paths. The OneLake side is a shortcut, not a copy.

When you "Link to Fabric" from Dataverse, you're not copying data. You're creating a shortcut from OneLake into Dataverse's underlying storage. Dataverse remains the source of truth. OneLake reads through to Dataverse storage at query time.

Where the data lives

The data is physically stored by Dataverse, in Azure storage that Dataverse manages. OneLake holds a pointer.

How updates flow

A record updated in Dataverse appears in OneLake within ~15 minutes (Fast Fabric). No pipeline to maintain.

How writes flow back

Fabric can compute results, then expose them to Dataverse as virtual tables — Dataverse tables that read from OneLake on demand.

Watch for region mismatches. Link to Fabric requires the Dataverse environment and the Fabric workspace to be in the same Azure region. Cross-region linking has historically been blocked or required Microsoft engagement. Azure Synapse Link for Dataverse (the predecessor) has more flexibility here, which is why some teams have stayed on it.
What about data ingested directly into Fabric?

If you upload a CSV to a Lakehouse, or a pipeline pulls from an external API into OneLake, then Fabric IS the source of truth for that data. "Dataverse stays the source" only applies to data that originates in Dataverse.

Integration

Mirroring

Continuous replication of an external database into OneLake. The right tool for your Postgres question.

Mirroring continuously replicates an external database into OneLake as Delta tables. Changes in the source database flow into OneLake in near real-time.

It's different from a shortcut. A shortcut points at the original storage and reads through. A mirror creates copies in OneLake that stay synchronized with the source.

Currently supported sources for mirroring include:

  • Azure SQL Database
  • Azure Cosmos DB
  • Snowflake
  • PostgreSQL (Azure Database for PostgreSQL)
  • Other databases via "Open Mirroring," which lets you push your own change data

Your Postgres scenario

The question: Postgres database. Want it available in OneLake. Is mirroring the answer?

Short answer: Yes, if it's Azure Database for PostgreSQL or you can route through Open Mirroring.

What you get:

  • Postgres tables appear in OneLake as Delta tables
  • Updates in Postgres flow into OneLake automatically
  • Power BI, notebooks, and warehouses can query the OneLake copy
  • Postgres operational queries don't get impacted by analytics load

What it doesn't do:

  • Mirroring is one-way (Postgres → OneLake only)
  • Writes to OneLake mirrors don't go back to Postgres
The naming problem

The dataflow confusion

Three different things share the name "dataflow." This is the source of most documentation pain.

Dataflow Gen1

Power BI · Legacy

The original Power BI dataflow. A Power Query workload that lands data in the Power BI service for use in semantic models.

Where
Power BI workspace
Output
Common Data Model (CDM) folders in ADLS
Engine
Power Query Online
Use for
Existing Power BI work that hasn't migrated yet

Dataflow Gen2

Fabric · Data Factory

The Fabric replacement. Same Power Query interface, but runs on Fabric's SQL compute engine and writes directly to OneLake.

Where
Fabric workspace (Data Factory)
Output
Lakehouse, Warehouse, KQL DB, or Azure SQL
Engine
Power Query + Fabric SQL compute
Use for
New transformation work in Fabric

Dataverse Dataflow

Power Platform

A separate feature for loading data into Dataverse tables. Also Power Query under the hood, but the destination is Dataverse, not OneLake.

Where
Power Apps maker portal
Output
Dataverse tables
Engine
Power Query Online
Use for
Bulk-loading data into Dataverse

All three use Power Query as the transformation engine. They differ in where they live and where they write to.

Terms you'll hear and what they mean.
Capacity
The compute and storage allocation that powers your tenant. Billed in CUs.
Workspace
The security and collaboration boundary inside Fabric. Permissions are granted here.
Item
Anything you create in a workspace: Lakehouse, Warehouse, Report, Notebook, Pipeline.
Lakehouse
Workspace item that holds files and Delta tables. Spark-friendly entry point.
Warehouse
Workspace item that exposes Delta tables through T-SQL with full transactional support.
DirectLake
Power BI mode that reads Delta files in OneLake directly. Import speed, no refresh.
Shortcut
A pointer in OneLake to data stored elsewhere (S3, ADLS, Dataverse). No copy.
Mirroring
Continuous replication of an external database into OneLake as Delta tables.
Medallion
Pattern of organizing data in three layers: Bronze (raw), Silver (cleaned), Gold (curated).
Activator
Fabric tool that watches data and triggers actions when conditions are met.
Eventstream
Real-Time Intelligence's ingestion layer. Connects to streaming sources.
Eventhouse
The KQL database underneath Real-Time Intelligence.
Virtual table
A Dataverse table that reads from OneLake on demand. Bridges Fabric outputs to Power Apps.
Domain
A logical grouping of workspaces above the workspace level. Used in data mesh patterns.
Sources: Microsoft Learn (learn.microsoft.com/fabric), Power Platform docs
